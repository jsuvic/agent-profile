# Subagent Execution Policy

`subagentPolicy` is opt-in. When it is omitted or `enabled: false`, generated
files and lockfile provenance remain byte-identical to the corresponding
profile without the policy.

```yaml
subagentPolicy:
  enabled: true
  roles:
    architect:
      capability: strongest
      effort: extra-high
      overrides:
        codex:
          model: gpt-5.2-codex
          effort: extra-high
        claude:
          model: claude-opus-4-1-20250805
          effort: high
  orchestration:
    maxConcurrentThreads: 3
    maxDepth: 1
    parallelWrites: false
  context:
    handoff: task-capsule
    memory: targeted
    indexed:
      mode: preferred
      provider: cce
  evidence:
    summary: required
    localTrace:
      enabled: false
      retention: 20
```

The canonical role values are provider-neutral. Exact model identifiers and
target effort are exceptions that must use the versioned allowlist described in
[Subagent Model Mapping v2 Evidence](../research/010-subagent-model-mapping-v2.md).
Codex and Claude receive resolved model guidance; Tabnine receives only portable
task-capsule and local-first conventions.

`indexed.mode: off` omits indexed-first guidance and explicitly directs bounded
native discovery. It neither installs nor indexes CCE.

## Mapping-v3 opt-in (`preset`)

An additive, optional `preset` field opts a profile into the mapping-v3
model-policy resolver instead of the mapping-v2 allowlist above:

```yaml
subagentPolicy:
  enabled: true
  preset: role-aware # or: quality-first | cost-conscious
  roles:
    routine-implementer:
      capability: balanced
      effort: medium
```

- `preset` is one of `role-aware | quality-first | cost-conscious`. Omitting
  it retains mapping-v2 behavior byte-for-byte; see
  [Model Selection Lifecycle](../specs/phase-31.5/001-model-selection-lifecycle.md)
  for the preset tables.
- `routine-implementer` is a v3-only role: it is accepted alongside the
  existing nine roles but has no mapping-v2 legacy resolution, so it is only
  meaningful on a profile that also sets `preset`.
- When `preset` is set, `overrides.codex.model` / `overrides.claude.model`
  accept any non-empty, control-character-free string under 200 characters —
  not only the pinned mapping-v2 allowlist above. An uncatalogued exact
  identifier is accepted and resolves `unverified`/unrated rather than being
  rejected; see the parent spec's target-capability-status contract.
- Without `preset`, `overrides.codex.model` / `overrides.claude.model` are
  still restricted to the pinned mapping-v2 allowlist.
- I2 carries those validated Codex/Claude exact overrides into generated
  guidance, the owned project-local Codex primary default, and lockfile
  provenance. Tabnine remains a separate Phase 31.5 issue.

## Mapping-v3 Codex/Claude exact catalog and generated artifacts (Phase 31.5 I2)

When `preset` is set, generated Codex and Claude artifacts (the shared
`## Subagent Execution Policy` table in `AGENTS.md`, imported by `CLAUDE.md`;
and the project-local `.codex/config.toml` primary default) resolve against
the mapping-v3 exact target catalog instead of the mapping-v2 allowlist. A
profile without `preset` (or with `subagentPolicy` disabled) keeps every
generated byte identical to the pre-I2 output; there is no partial migration.

Evidence: [Model Policy Mapping v3 Evidence](../research/012-model-policy-mapping-v3-evidence.md)
(verified 2026-07-16).

### Preset and explicit role override precedence

Per the parent spec's "Model presets" contract ("Explicit role and target
overrides continue to win over these defaults") and Decision Rule 4 ("role
entries remain explicit overrides over that preset"): when a v3-opted profile
sets BOTH `preset` and an explicit `subagentPolicy.roles[id].capability` /
`.effort` for a role, that role's explicit `capability`/`effort` wins over
`MODEL_POLICY_PRESET_TABLE[preset][role]`. A role that is absent from `roles`
(the common case for a v3-opted profile) resolves the selected preset's own
row unchanged.

**All three v3 surfaces honor this precedence identically and cannot
disagree for the same profile**: the shared `AGENTS.md`/`CLAUDE.md` guidance
table, the project-local `.codex/config.toml` primary-default write (when the
overridden role is `implementer`, the designated primary role), and
`ai-profile.lock`'s `modelPolicy` block all derive their per-role
capability/effort through one single-owner conversion
(`deriveModelPolicyRoleOverrides` in `model-policy-target-adapter.ts`) before
resolving against the Codex/Claude catalogs. There is no path by which the
guidance table could claim a role is `configured` with one model while
`.codex/config.toml` or the lockfile records a different one for that same
role. For example:

```yaml
subagentPolicy:
  enabled: true
  preset: role-aware
  roles:
    explorer:
      capability: strongest
      effort: extra-high
```

Here `explorer` renders `strongest`/`extra-high` (Codex Sol/`xhigh`, Claude
Fable 5/`xhigh`) in the guidance table instead of `role-aware`'s own
`efficient`/`low` row for that role; every other role still resolves from
`role-aware` unchanged. Overriding `implementer` (the primary-default role)
the same way changes what `.codex/config.toml` and `ai-profile.lock` actually
record too — see
[`model-policy-lockfile-wiring.test.ts`](../../packages/compiler/src/model-policy-lockfile-wiring.test.ts)
for a regression test asserting all three surfaces agree. This precedence
also covers exact target values. Exact
`roles[id].overrides.codex.model` / `.claude.model` values take precedence
over the role's catalog candidate. A catalogued value records its catalog
lifecycle; an uncatalogued organization/private value is kept verbatim,
marked `unrated` and `unverified`, and is never silently replaced. Exact
overrides record `source: explicit-override`, have no invented alternatives,
and flow through the same shared row used by guidance, the Codex primary
configuration (for `implementer`), and the lockfile.

### Per-surface capability status

A v3-opted profile computes capability status separately per role, client,
and surface — the same role/client pair can be `configured` on one surface
and `advisory` on another:

- **Primary-default** (the single project-local surface Agent Profile
  actually writes): only the `implementer` role's Codex resolution is written
  into `.codex/config.toml` (`model` / `model_reasoning_effort`), so that row
  reports `configured`. Every other role's Codex primary-default status is
  `advisory` (Codex documents per-agent `model`/`model_reasoning_effort` and
  interactive `/model`, but Agent Profile does not drive those surfaces).
  Claude has no documented project-local file Agent Profile writes to select
  an exact model/effort today, so Claude's primary-default status is always
  guidance-only.
- **Per-workflow/skill** (the shared `AGENTS.md`/`CLAUDE.md` role table):
  `advisory` for confirmed-official identities (Codex Sol/Terra/Luna; Claude
  Opus 4.8, Haiku 4.5).
- **Per-subagent**: the same guidance-only status as per-workflow/skill;
  Agent Profile does not currently write an exact per-subagent model into
  `.claude/agents/*.md` (`model: inherit`) or `.codex/agents/*.toml`.
- **Unverified override**: any role/surface whose resolved Claude model is
  `claude-fable-5` or `claude-sonnet-5` reports `unverified` on every
  surface, overriding what would otherwise be `configured`/`advisory` — their
  Claude Code effort/frontmatter behavior is `client-verification-required`
  per the pinned evidence note, not a completed implementation fact.
  An uncatalogued exact Codex or Claude override is likewise `unverified` on
  every surface because representability does not prove client behavior.
- **Unsupported**: a role whose preset capability has no ordinary catalog
  candidate reports `unsupported` on every surface (not expected for the
  current Codex/Claude v3 catalogs, which cover all three capabilities).

### Lockfile provenance

`ai-profile.lock`'s `modelPolicy` block (schema already shipped in I1) is
populated only for a v3-opted profile, using `catalogVersion: 3` and one
resolution row per role/client pair that resolved a model. Each row's
`capabilityStatus` is the primary-default status for the `implementer`/Codex
row and the per-workflow/skill status for every other row. The `effort` field
stores the target-native emitted value (`xhigh`, not canonical `extra-high`,
for current Codex/Claude adapters), so provenance agrees with target controls
and guidance. See
[`fixtures/subagent-policy-v3-role-aware-enabled/`](../../fixtures/subagent-policy-v3-role-aware-enabled/)
for a full golden example, and
[`fixtures/subagent-policy-enabled/`](../../fixtures/subagent-policy-enabled/)
for the mapping-v2 byte-identity regression case (no `preset`, no
`modelPolicy` block).

### Explicit non-goals of this phase

- Live availability probing of an installed Codex/Claude client (Phase 31.5
  I4).
- Global/user configuration outside the project-local files above.
- Runtime fallback orchestration between the primary model and its listed
  alternatives, or any credential/provider configuration.
- Tabnine's mapping-v3 catalog (Phase 31.5 I3).
