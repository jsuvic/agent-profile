# Spec: Compiler Determinism

## Status

Verified

Implemented in `packages/compiler` for the approved `agents-md`,
`lockfile`, `tabnine-guidelines`, `tabnine-mcp-config`, `codex-config`,
`claude-settings`, and `claude-mcp` targets. Verified on 2026-05-02 after final
implementation review.

## Problem

Agent Profile Compiler must produce the same generated bytes for the same
validated profile, target set, template set, and compiler version. Any
nondeterminism makes golden tests noisy, weakens lockfile drift detection, and
undermines user trust.

## Goal

Define the deterministic compiler API and output contract shared by all output
targets.

## Non-Goals

- implementing target-specific templates
- implementing CLI commands
- implementing lockfile writes
- implementing doctor checks
- supporting concurrent writes
- adding hosted compilation

## User Flow

1. A caller provides a validated `AiProfile` from `001-profile-schema-v1.md`.
2. A caller requests one or more targets.
3. The compiler selects templates for those targets.
4. Each target returns generated file descriptors.
5. The compiler normalizes and sorts descriptors before diff, write, lockfile,
   and golden test usage.

## Inputs

```ts
type CompileRequest = {
  profile: AiProfile;
  targets: CompilerTargetId[];
  templates: TemplateDescriptor[];
};

type CompilerTargetId =
  | "agents-md"
  | "lockfile"
  | "tabnine-guidelines"
  | "tabnine-mcp-config"
  | "codex-config"
  | "claude-settings"
  | "claude-mcp";
// Forward reference (subagents): docs/specs/phase-11/ adds
// "claude-subagents", "codex-subagents", and "tabnine-subagents" to this
// union when Phase 11 is approved. The Codex subagents target also amends
// the existing "codex-config" template's deterministic expansion to append
// an additive [agents] block; the target id does not change.

type TemplateDescriptor = {
  id: string;
  target: CompilerTargetId;
  version: string;
  sha256: string;
};
```

## Outputs

```ts
type CompileResult =
  | {
      ok: true;
      files: GeneratedFile[];
      templates: TemplateDescriptor[];
    }
  | {
      ok: false;
      issues: CompileIssue[];
    };

type GeneratedFile = {
  path: string;
  target: CompilerTargetId;
  templateId: string;
  bytes: Uint8Array;
  sha256: string;
};
```

## Failure Modes

```ts
type CompileIssue = {
  code:
    | "unsupported_target"
    | "disabled_target"
    | "missing_template"
    | "invalid_output_path"
    | "nondeterministic_output"
    | "unsafe_generated_content";
  path: string;
  expected: string;
  actual: string;
  message: string;
};
```

Rules:

- `path` is a target ID, generated output path, or JSON Pointer into the request.
- Issues are sorted by `path`, then `code`, then `message`.
- Issue messages must not include source contents, secret values, or environment
  variable values.

## Contracts

- The compiler must not read the clock, random values, process IDs, local user
  names, absolute workspace paths, environment variable values, network state,
  or Git state for generated output.
- Output paths are repository-relative and use forward slashes.
- Output paths must not contain `..`, drive letters, backslashes, empty
  segments, or absolute path prefixes.
- Output ordering is lexicographic by `path`, then `target`.
- Generated text uses LF line endings.
- Generated text ends with exactly one trailing newline.
- Generated text is encoded as UTF-8 bytes before hashing.
- `sha256` hashes the exact generated bytes.
- Template versions are trackable by stable IDs and hashes.
- Same inputs must produce byte-identical `GeneratedFile.bytes` and identical
  hashes.
- The compiler returns generated descriptors only; writing files is a future CLI
  responsibility.

## Normalization Rules

- Convert CRLF and CR line endings to LF.
- Trim trailing whitespace from generated blank lines.
- Ensure exactly one trailing newline at end of text files.
- Preserve intentional indentation inside Markdown, TOML, JSON, and skill files.
- Do not include generation timestamps or absolute paths.

## Security Rules

- Do not upload source code or profile contents.
- Do not read secret files.
- Do not include literal secrets in generated output.
- Do not include environment variable values in generated output.
- Do not execute shell commands.
- Do not install dependencies.
- Do not generate broad auto-approved tool permissions.
- Mutating tools and shell-capable flows must default to ask/deny unless an
  approved target spec says otherwise.

## Acceptance Criteria

- A compiler API contract exists for deterministic generated files.
- Output ordering, path normalization, line endings, encoding, and trailing
  newline behavior are defined.
- Template metadata required for `002-lockfile-v1.md` is defined.
- Stable compile issue codes and issue ordering are defined.
- Determinism constraints are testable without file writes.

## Tests

- same request produces byte-identical outputs twice
- output paths are sorted by `path`, then `target`
- generated text uses LF only
- generated text has exactly one trailing newline
- generated output hashes match generated bytes
- invalid output path fails
- unsupported target fails
- disabled target fails when explicitly requested
- missing template fails
- output does not include absolute workspace paths
- output does not include environment variable values
- output does not include secret-like literals
