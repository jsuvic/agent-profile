# I1: Future-configuration ownership decision

## Parent spec or request

`docs/specs/phase-32/001-guided-repository-update.md`

## Intent summary

Distinguish valid user configuration for a not-yet-supported capability from
corrupt generated output before compile, Doctor, or reconciliation chooses a
destructive or failing path.

## Behavior slice

A pure classifier consumes destination, parsed structural shape, generated
baseline identity, lock ownership, and canonical representability. It returns
generated-owned, user-owned-future-configuration, or invalid with a stable
reason. Phase 32 initially applies the new classification only to root
`.mcp.json`.

## Non-goals

- Filesystem mutation or lockfile writing.
- MCP declaration, merge, launch, installation, or runtime verification.
- Generalizing every generated destination without a later approved spec.

## Acceptance criteria

- Phase-32 acceptance criterion 1.
- Exact generated empty root `.mcp.json` remains generated-owned.
- Valid non-empty root MCP configuration that is not canonically representable
  becomes user-owned future configuration.
- Malformed JSON, unsafe shape, and unrelated destinations retain existing
  invalid/ownership behavior.
- The decision and reason are deeply immutable and deterministically ordered.

## Expected RED proof

The current ownership path classifies a valid custom root `.mcp.json` only as
generated byte drift and has no representability-aware decision.

## Expected GREEN proof

The full ownership table passes independently of filesystem orchestration, with
exact stable reasons and no changes to unrelated destination rows.

## Seam under test

`classifyFutureConfiguration(input) -> FutureConfigurationOwnershipDecision`.

## Allowed mock boundary

None. Use literal metadata/byte fixtures; parsing and classification are owned
code and remain real.

## Test command guidance

Run the focused compiler ownership table, then compiler tests, check, lint, and
verify:pack.

## Likely file ownership

- Compiler/import ownership classification modules
- Shared lock/reconciliation ownership types
- Focused pure table fixtures

## Dependencies

`sequenced` after Phase 31.5 I9.

## Parallelism notes

Parallel-safe with Phase 32 I3 and I4 after Phase 31.5. Own the shared ownership
decision before I2 consumes it.

## Contract impact

Adds one scoped ownership state/reason. Exact generated and unrelated ownership
contracts remain unchanged.

## Security impact

Classify structure without environment values, secret reads, server launch,
network, or filesystem writes.

## Documentation impact

ADR 0020 terminology and ownership-state reference.

## Implementation context

Keep canonical representability explicit. Do not infer user intent merely from
byte inequality or introduce a structural merge policy.

## Review expectations

Require exhaustive decision rows, deep immutability, stable reasons, and proof
that malformed or unrelated content does not enter the new state.
