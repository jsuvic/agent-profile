## Status

Phase 10.5 later advanced only `tdd-change` to template `@2`.

## Problem

The MVP plan calls for generated workflow skills for Claude, but the repository
only has the doctor skill checks. Without a target spec, the compiler cannot
safely generate task-specific SDD, TDD, and final-review skills under Claude
Code's documented project skill path.

## Goal

Generate deterministic project-scoped Claude workflow skills from
`ai-profile.yaml` workflow intent.

The MVP skill set is:

- `sdd-change`
- `tdd-change`
- `final-review`

## Non-Goals

- implementing the target
- generating Codex skills
- generating Claude plugins
- generating `.claude/commands/`
- generating `.claude/rules/`
- generating scripts, references, templates, or assets
- generating dynamic context injection commands
- pre-approving tools with `allowed-tools`
- changing Claude settings, permission modes, local settings, or managed
  settings
- generating user-level, managed, or plugin skills

## User Flow

1. A user enables `clients.claude.enabled: true`.
2. A user enables one or more workflow flags in `ai-profile.yaml`.
3. A future compile command renders project-local Claude skills under
   `.claude/skills/`.
4. Claude Code discovers the project skills and can invoke them explicitly with
   `/skill-name` or automatically from the description.
5. Golden tests compare every generated skill file byte-for-byte.
6. Doctor checks skill size and trigger descriptions.

```markdown
<!-- Generated by Agent Profile Compiler. Do not edit by hand. -->

# TDD Change

## Instructions

1. Identify the smallest observable behavior covered by the approved spec.
2. Add or update one focused failing test or golden fixture before changing behavior code.
3. Run the narrowest relevant test command and confirm RED: the test fails for the expected reason, not because of a typo, setup error, or unrelated failure.
4. Implement the smallest change that satisfies the failing test and the spec.
5. Run the same focused test command and confirm GREEN: the test passes without new warnings or unrelated failures.
6. Refactor only after GREEN, then rerun the focused test command.
7. Do not update golden files only to hide an unexplained behavior change.

## Testing Anti-Patterns

- Do not assert on mock elements or mock call counts when a real behavior assertion is possible.
- Do not add production methods, flags, or exports that exist only for tests.
- Do not mock a dependency until you understand the side effects the test needs.
- Keep test doubles structurally complete enough to match the real data shape consumed by the code.
- If mock setup is larger than the behavior under test, consider a narrower integration test or a simpler production boundary.

## Output

Report the RED command and expected failure, the GREEN command and passing result, any refactor rerun, and any TDD exception that required human approval.

## Safety

- Do not install dependencies automatically.
- Do not run broad or destructive commands without explicit approval.
- Do not include secrets, environment variable values, or production endpoints in tests or fixtures.
```

## Target Mapping

Verified Claude Code skills surface for this spec revision:

Verification date: 2026-05-03.

- Claude Code skills use `SKILL.md` files with YAML frontmatter and Markdown
  instructions.
- Project skills live at `.claude/skills/<skill-name>/SKILL.md`.
- The directory name becomes the slash command name.
- The description helps Claude decide when to load a skill automatically.
- Project skills can also be invoked directly with `/skill-name`.
- `description` is recommended and should put the key use case first.
- Optional fields such as `allowed-tools`, `disable-model-invocation`,
  `context`, `agent`, `argument-hint`, `arguments`, `model`, and `effort` exist
  but are outside this MVP target unless explicitly listed in this spec.
- `allowed-tools` grants listed tools without per-use approval while a skill is
  active, so this MVP target must not emit it.
- Dynamic context injection with shell execution exists, so this MVP target
  must not emit `!` shell placeholders or fenced shell-injection blocks.

Verified source URL:

- `https://code.claude.com/docs/en/skills`

Mapping rules:

| Profile data           | Claude skill output                        |
| ---------------------- | ------------------------------------------ |
| `workflow.sdd: true`   | `.claude/skills/sdd-change/SKILL.md`       |
| `workflow.tdd: true`   | `.claude/skills/tdd-change/SKILL.md`       |
| `workflow.finalReview` | `.claude/skills/final-review/SKILL.md`     |
| `effectivePermissions` | safety wording inside each generated skill |
| disabled workflow flag | corresponding skill is not generated       |

Official-doc verification is captured as a required implementation gate:

- Before implementing this target, re-check current Claude Code skills docs.

  this spec, update this spec and golden fixtures before writing target code.
- Do not emit optional skill fields that affect tool permissions, shell
  execution, subagents, or model choice without a later approved spec.

## Contracts

- Generated skills consume `effectivePermissions` for safety wording.
- Generated skills must start with YAML frontmatter.
- Generated frontmatter must include `name` and `description`.
- Skill descriptions must include clear trigger/use-case language.
- The target must generate only project-local `.claude/skills/**/SKILL.md`
  files.
- The target must not generate `.claude/commands/` files.
- The target must not emit `allowed-tools`.
- The target must not emit shell dynamic context injection.
- The target must not include scripts, references, templates, assets, plugins,
  or subagent configuration.
- Each generated skill must stay below the 300-line doctor warning threshold.
- Golden tests compare exact output bytes.

## Security Rules

- Do not include literal secrets.
- Do not include environment variable values.
- Do not instruct source upload.
- Do not grant production access.
- Do not instruct automatic dependency installation.
- Do not instruct unsafe auto-approval.
- Do not grant tool pre-approval with `allowed-tools`.
- Do not use dynamic context injection or bundled scripts.
- Do not generate user, managed, or plugin skill artifacts.
