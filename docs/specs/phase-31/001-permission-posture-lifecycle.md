# Spec: Permission Posture Lifecycle

## Status

Approved on 2026-07-14 from the synthesized permission-control agreement.
Governed by ADRs 0002, 0004, 0005, 0014, and 0019 as amended by this phase.

## Problem

Agent Profile Compiler generates restrictive client permission settings but
does not give users one coherent lifecycle for choosing, inspecting, changing,
activating, and validating the intended level of agent control. Client-native
settings can merge across project, local, user, managed, and session scopes, so
a higher-precedence scalar preference may still be defeated by lower-scope
permission arrays or one-way restriction flags.

The current result is especially confusing for existing repositories: a user
can intentionally configure a more autonomous local client posture while
generated shared settings continue to force prompts, and doctor reports the
intentional local change as unsafe drift rather than reconciling it with a
declared product intent.

## Goal

Add a complete, local-first permission-posture lifecycle that:

- offers three normal development postures: `guarded`, `balanced`, and
  `trusted-local`;
- preserves `plan-only` for audit/review work;
- preserves existing `autonomous` profiles and generated bytes as a legacy,
  sandbox-required contract until the user explicitly migrates;
- supports a global baseline plus optional client-specific posture choices;
- inspects declared and detectable actual posture without reading secrets;
- guides repair, adoption, personal activation, and manual client setup;
- maps only verified client capabilities and reports limitations honestly; and
- makes doctor ownership-aware and risk-asymmetric.

## Intent

The user, not the compiler, owns the risk decision. APC should make that
decision understandable, deterministic, reviewable, and reversible without
silently granting authority or forcing every target to the lowest common
denominator.

## Decision Rules

1. Outcome language before client syntax: users choose a posture and review the
   resulting client mappings before seeing raw client keys.
2. Safety intent before local drift: shared profile intent is canonical, but
   detectable existing behavior may be explicitly adopted when it is
   representable without loss.
3. Hard denial before autonomy: secret access, source upload, production access,
   and telemetry remain denied regardless of posture or client adjustment.
4. Personal consent before high autonomy: shared files may make a chosen posture
   possible, but `trusted-local` activation is developer-local and separately
   confirmed.
5. Verified capability before generation: unknown or manual-only client
   behavior is reported, never guessed.
6. Risk asymmetry before byte equality: looser-than-declared behavior is a
   safety error; stricter-than-declared behavior is a usability warning.
7. Existing bytes before migration convenience: omission of the new posture
   contract and legacy `autonomous` behavior remain byte-identical until an
   explicit interactive migration.
8. Shared state machinery before new detection: the bare dispatcher,
   configure flow, compile planner, and doctor consume one canonical posture
   plan rather than reimplementing permission interpretation.

## Architecture Rescue Candidate

### Canonical permission-posture plan

- Files/modules involved: core profile normalization and
  `deriveEffectivePermissions`, compiler target renderers, doctor permission
  evaluation, CLI dispatcher/configure orchestration, and the local UI profile
  model.
- Current friction: safety mode, effective permission derivation, target
  mapping, doctor interpretation, and CLI setup defaults are spread across
  separate modules and can disagree about the same profile.
- Proposed interface: one immutable, client-neutral
  `PermissionPosturePlan` produced from validated profile intent. It contains
  baseline posture, client posture, effective granular permissions, hard
  denials, legacy status, and target mapping status.
- Locality/leverage: compiler, doctor, dispatcher, configure, and UI consume
  the same plan; client adapters own only documented native mappings.
- Test improvement: one table-driven pure seam freezes precedence and legacy
  compatibility before any target or orchestration change.
- ADR/spec conflicts: amends ADRs 0002 and 0004; must preserve old profile and
  golden bytes until explicit adoption.
- Dependency state: prerequisite (Phase 31 I1).

## Non-Goals

- Weakening hard denials for secrets, source upload, production access, or
  telemetry.
- Silent posture adoption through upgrade, CI, JSON, quiet, piped, or other
  non-interactive flows.
- Changing byte-identical non-TTY bare-command help or running detection on
  that path.
- Managing remote or enterprise-enforced client policies.
- Reading or importing credentials, tokens, environment values, or unrelated
  client settings.
- Owning an entire user or local client settings file.
- Inventing undocumented Claude, Codex, or Tabnine settings.
- Claiming equivalent behavior across clients.
- Launching clients, executing hosted workflows, or verifying posture by
  performing risky actions.
- Requiring deterministic Tabnine configuration where official documentation
  exposes only a manual UI workflow.
- Adding clients beyond Claude, Codex, and Tabnine.
- Redesigning unrelated setup profiles, capability upgrades, or compilation
  behavior.

## User Flow

### Interactive bare command

Interactive `npx agent-profile` remains the state-aware router. Permission
control is always listed as `Change agent control`, with the current posture
preselected inside that flow. It becomes the recommended action only for first
setup, safety violations, posture/config drift, incomplete personal
activation, or legacy migration.

Non-TTY, piped, and CI bare invocation remains byte-identical help and performs
no detection.

### Explicit configure command

`agent-profile configure` opens the same state-aware configuration flow. The
flow:

1. Reads repository-scoped profile and client permission metadata.
2. Shows the current declared posture and detectable effective posture.
3. With explicit consent, inspects only known permission/sandbox fields in
   user- or machine-level local configuration.
4. Labels unobserved managed, remote, or session-only state as unknown.
5. Offers `guarded`, `balanced`, `trusted-local`, `plan-only`, and optional
   per-client adjustment; `autonomous` is shown only when preserving or
   migrating a legacy profile.
6. Shows the client mapping status and exact behavioral delta.
7. Offers repair to declared intent, adoption of representable detected intent,
   exact inspection, or no change.
8. Previews and atomically applies shared profile/generated changes.
9. Separately offers personal activation and manual client steps.
10. Reinspects and prints the resulting declared, configured, and unknown
    states.

### Existing autonomous setup

An existing `safety.mode: autonomous` profile is never reinterpreted. The guide
shows:

- Keep legacy Autonomous: retains sandbox-required semantics and byte-identical
  outputs.
- Migrate to Trusted local: previews the changed safety meaning, generated
  files, and personal activation requirements.
- Choose Guarded, Balanced, or Plan-only instead.

No migration choice is pre-applied. `upgrade --write --adopt-recommended` does
not adopt a posture.

### Reconciliation

When actual configuration differs from declared intent, the guide offers:

- Repair actual configuration to the declared posture.
- Adopt detected behavior as profile intent, only when losslessly
  representable.
- Review differences and configuration sources.
- Leave unchanged; doctor continues to report the mismatch.

Unrepresentable client behavior remains manually owned and is never
approximated into the canonical profile.

### Staged writes

1. Inspection and plan are read-only.
2. Shared `ai-profile.yaml` and generated artifacts are previewed and written
   atomically through existing planners.
3. Personal activation is offered only after the shared write succeeds.
4. A supported local activation writer changes only owned permission fields,
   preserves unrelated content, refuses unsafe structure/symlinks, and requires
   the local file to be ignored or an explicit ignore update.
5. Manual-only client instructions are presented after shared success.
6. Failed personal activation leaves the valid shared profile in place and is
   reported as incomplete activation.

## Inputs

- `ai-profile.yaml` safety mode, sandbox intent, granular permissions, enabled
  clients, and optional client posture adjustments.
- `ai-profile.lock` ownership and hashes.
- Generated project client settings.
- Repository-local client settings such as Claude's local project settings.
- Consent-gated, permission-fields-only views of known user/machine settings.
- Current official client capability mapping catalog with verification date.
- Interactive posture, reconciliation, inspection, and activation choices.

Proposed additive profile shape for the I1 human gate:

```yaml
safety:
  mode: trusted-local
  requiresSandbox: false

clients:
  tabnine:
    enabled: true
    permissionPosture: guarded
  codex:
    enabled: true
    permissionPosture: inherit
  claude:
    enabled: true
    permissionPosture: trusted-local
```

The schema continues to accept legacy `autonomous`; new setup does not offer it
as a normal choice. Exact field names and validation codes are frozen by I1 RED
tests before implementation.

## Outputs

- Immutable canonical `PermissionPosturePlan`.
- Deterministic shared Claude/Codex artifacts where officially supported.
- Capability-accurate Tabnine guidance and manual activation instructions.
- Configure preview and apply report separated into shared and personal stages.
- Permission inspection evidence containing setting names, postures, source
  scopes, mapping status, and confidence only.
- Doctor findings with ownership-aware, risk-asymmetric severity.
- Updated schema/docs/examples/goldens/lockfile provenance where required.

Mapping statuses are a closed set:

```text
configured-automatically
personal-activation-required
manual-setup-required
unsupported
blocked-by-policy
unknown
```

## Contracts

- Profiles that do not explicitly adopt the new contract compile
  byte-identically.
- Legacy `autonomous` profiles remain valid, sandbox-required, and
  byte-identical until explicit migration.
- `plan-only` remains supported and is not collapsed into a normal development
  posture.
- Normal development posture values are `guarded`, `balanced`, and
  `trusted-local`.
- Baseline posture supplies defaults; a client adjustment replaces only that
  client's baseline; explicit global granular permissions and hard denials
  remain authoritative.
- Secret and production permission dimensions remain deny-only in schema v1.
- Shared generated settings must not contradict the selected posture.
- Trusted-local activation is never silently shared or silently applied to a
  developer's machine.
- Permission inspection reads known permission/sandbox metadata only and never
  claims unobserved scopes are safe.
- Permission arrays/scopes are evaluated using the target's documented merge
  and precedence rules.
- Adoption is refused when detected behavior cannot be represented without
  loss.
- Shared writes use existing preview/consent/atomic-write machinery.
- Personal activation is a separate consent and write boundary.
- The bare command remains a router; explicit subcommands remain first-class.
- Non-TTY bare output, JSON/quiet contracts, and existing exit codes remain
  frozen unless this spec explicitly adds a new configure-only contract.
- Client mapping data records source URL, verification date, support grade,
  and mapping version.

### Doctor issue contracts

- `LINT-PERM-003`: hard secret/production denial weakened (`error`).
- `LINT-PERM-004`: dangerous posture violates its declared sandbox/hard-safety
  contract (`error`). Legacy Autonomous retains its existing sandbox rule.
- `LINT-PERM-005`: actual/generated behavior is looser than declared (`error`)
  or an agent-profile-owned artifact differs unexpectedly (`error`, with
  ownership/drift codes retained where applicable).
- `LINT-PERM-006`: effective runtime state cannot be verified (`warning`); the
  result must not be summarized as aligned.
- `LINT-PERM-007`: effective posture is stricter than declared or personal
  activation is incomplete (`warning`).
- `LINT-PERM-008`: documented manual/unsupported target limitation (`info`).
- A confirmed personal activation matching declared intent passes.
- A legacy profile without migration passes with an informational migration
  offer.

## Security Rules

- No source-code, generated-config, or secret upload.
- No reading `.env`, credential stores, tokens, environment values, hook
  payloads, or unrelated settings.
- No telemetry or hosted execution.
- No client invocation for runtime verification.
- No dependency or MCP installation.
- No automatic enterprise/managed policy changes.
- No global/user file mutation unless a later client-specific spec explicitly
  authorizes a bounded personal activation writer; this phase only authorizes
  a project-local, ignored Claude permission-field writer.
- No local activation write through a symlink or unsafe/unknown structure.
- No posture or client adjustment may weaken hard denials.
- Findings and reports contain setting names and normalized states, never
  secret-like values or unrelated config content.

## Acceptance Criteria

1. The canonical resolver supports Guarded, Balanced, Trusted local,
   Plan-only, client adjustments, granular overrides, and immutable hard
   denials with a table-driven precedence contract.
2. Omitted new fields and every existing legacy fixture compile
   byte-identically; existing Autonomous fixtures remain sandbox-required and
   byte-identical.
3. Interactive setup/configure shows current posture, all applicable choices,
   resolved client outcomes, hard denials, mapping statuses, and preview before
   any write.
4. Existing Autonomous setup offers keep, explicit Trusted-local migration,
   other posture selection, and cancel; no branch migrates silently.
5. Claude shared generation for Trusted local removes contradictory routine
   prompt requirements while preserving hard denials; local activation is a
   separate, ignored-file, permission-field-only write.
6. Codex mappings cover documented sandbox/approval combinations and report
   personal activation as manual/session/profile work when no safe project-local
   writer exists.
7. Tabnine mappings report per-tool IDE setup and CLI limits without generating
   undocumented settings.
8. Permission inspection evaluates repository scopes automatically, asks
   before user/machine inspection, reads only approved fields, identifies every
   source, and reports unavailable managed/session state as unknown.
9. Reconciliation offers repair/adopt/review/leave; adoption is lossless or
   refused; cancel and refusal leave all bytes unchanged.
10. Shared apply is atomic. Personal activation occurs only after shared
    success, requires separate confirmation, preserves unrelated local fields,
    and reports partial failure without rolling back valid shared intent.
11. Doctor emits the binding severity matrix above and never summarizes unknown
    state as aligned.
12. Interactive bare dispatch always lists Change agent control, recommends it
    only for agreed trigger states, and preserves existing command priority for
    unrelated states.
13. Non-TTY/piped/CI bare invocation is byte-identical and proves no detection,
    permission inspection, or presentation loading occurred.
14. Official Claude, Codex, and Tabnine mappings are reverified at implementation
    time and unsupported behavior remains capability-graded rather than
    invented.

## Tests

- Table-driven canonical resolver tests for every baseline, client adjustment,
  granular override, hard denial, and precedence collision.
- Legacy compatibility goldens for omitted fields, Autonomous, and Plan-only.
- Client mapping table tests and exact generated-output goldens per supported
  target/posture.
- Claude merge/precedence and project/local activation fixtures.
- Codex project/user/session precedence fixtures and unsupported personal-local
  activation cases.
- Tabnine manual/unsupported mapping fixtures.
- Permission-only inspection tests with filesystem/source-consent sentinels and
  forbidden-key traps.
- Reconciliation fixture matrix: repair, adopt, unrepresentable refusal,
  inspect, leave, cancel.
- Shared atomic-write and personal partial-failure tests.
- Doctor table for every ownership/posture/mapping-status severity row.
- Dispatcher priority, always-visible action, follow-up, cancel, and non-TTY
  byte-identity/no-detection sentinels.
- Runtime sentinels proving no secret files, environment values, client
  processes, network, telemetry, or hosted execution are touched.
- Packaging and published-CLI tests for `npx agent-profile` and
  `agent-profile configure`.

## TDD Strategy

I1 is a pure computation seam: validated profile -> immutable posture plan.
I2 is a deterministic generator seam: posture plan -> shared artifacts and
mapping report. I3 is a pure computation seam: inspected permission metadata ->
effective state and reconciliation choices. I4 is an orchestration seam:
configure choices -> preview/shared filesystem effect. I5 is an orchestration
seam: confirmed personal activation -> bounded local effect/manual report. I6
is a pure/reporting seam: intent + evidence + ownership -> doctor findings. I7
is an orchestration seam: repository state -> dispatcher choice/routed flow.
I8 closes the published end-to-end journey and documentation. One slice equals
one seam and one focused RED; owned code is never mocked.
