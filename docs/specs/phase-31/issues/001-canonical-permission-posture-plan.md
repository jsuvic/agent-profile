# I1: Canonical permission posture plan and compatibility seam

## Parent spec or request

`docs/specs/phase-31/001-permission-posture-lifecycle.md`

## Intent summary

Create one client-neutral, immutable posture plan so compiler, doctor,
configure, dispatcher, and UI cannot interpret the same profile differently.

## Behavior slice

A validated profile resolves to a `PermissionPosturePlan` containing baseline,
per-client posture, effective granular permissions, hard denials, legacy
status, and deterministic mapping inputs. Existing profiles and legacy
Autonomous profiles preserve current behavior and bytes.

## Non-goals

- Rendering client files.
- Reading client configuration.
- Adding CLI prompts or writes.

## Acceptance criteria

- Phase-31 acceptance criteria 1-2.
- Exact additive schema field names, closed enums, validation errors, and
  precedence are frozen in table-driven RED tests.
- `autonomous` remains accepted only with its existing sandbox contract;
  `trusted-local` is distinct; `plan-only` remains supported.

## Expected RED proof

New schema fixtures reject Trusted local/client posture fields and no canonical
resolver can produce the agreed plan or precedence rows. All old fixture
goldens remain green during RED.

## Expected GREEN proof

Schema/parser/resolver tables pass, returned plans are deeply immutable, and
all omission/legacy golden sentinels remain byte-identical.

## Seam under test

`resolvePermissionPosture(validatedProfile) -> PermissionPosturePlan`.

## Allowed mock boundary

None. This is pure deterministic computation; do not mock parser or resolver.

## Test command guidance

Run focused schema/core tests, then compiler compatibility goldens, full tests,
check, lint, and verify:pack sequentially.

## Likely file ownership

- `packages/schemas/ai-profile.schema.json`
- Core profile types/normalization/effective-permission derivation
- Schema/core fixtures and compatibility goldens
- Schema reference and ADR amendments

## Dependencies

`ready`: Phase 31 spec and ADR amendments were approved on 2026-07-14.

## Parallelism notes

Architecture-rescue prerequisite. Complete before I2-I7; owns shared canonical
types and schema.

## Contract impact

Additive only after explicit adoption. Omitted fields and legacy Autonomous
must preserve existing behavior and exact generated bytes.

## Security impact

Hard denials are immutable and authoritative. Reject any posture/override that
weakens secrets or production denial.

## Documentation impact

Schema reference, example profile, ADRs 0002/0004, Phase 31 terminology.

## Implementation context

Prefer a new core posture module consumed by existing
`deriveEffectivePermissions`; do not add parallel derivation in compiler,
doctor, or CLI. Deep-freeze nested outputs.

## Review expectations

Audit every precedence row, invalid enum, legacy input, omitted-field byte
sentinel, and hard-denial collision. Static typing alone is insufficient.
