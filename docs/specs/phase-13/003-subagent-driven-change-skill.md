# Spec: Subagent-Driven Change Skill

## Status

Approved and implemented. Belongs to Phase 13. Depends on
`001-subagent-template-reference-schema.md` and
`002-implementation-review-subagent-templates.md`.

## Problem

Generating implementation and reviewer subagents is not enough. The parent
agent also needs deterministic project-local guidance for when to use them,
what context to pass, which status values to expect, and how to sequence the
reviews.

Without an orchestration skill, a parent agent may delegate with vague context,
skip the spec-compliance review, run code-quality review too early, or accept
an optimistic worker report without independent inspection.

## Goal

Generate a `subagent-driven-change` workflow skill for Codex and Claude when
`workflow.subagentDrivenDevelopment: true`.

The skill defines:

- preconditions for delegation
- fresh-context prompting rules
- implementation worker dispatch expectations
- status handling
- two-stage review order: spec compliance first, code quality second
- final handoff requirements

## Non-Goals

- executing subagents during compile or doctor
- making the compiler orchestrate runtime subagent calls
- generating Tabnine workflow skills
- creating worktrees or branches
- committing, pushing, or opening pull requests
- installing dependencies
- adding scripts, references, assets, dynamic shell context, or tool
  pre-approval

## User Flow

1. The user enables `workflow.subagentDrivenDevelopment: true`.
2. The user references the three required templates:
   `implementer`, `spec-reviewer`, and `code-quality-reviewer`.
3. The compiler emits the parent orchestration skill for enabled supported
   clients.
4. At runtime, the parent agent loads the skill before delegating work.
5. The parent agent passes full task context to each subagent instead of
   relying on hidden conversation history.
6. The parent agent fixes or escalates spec-review findings before requesting
   code-quality review.

## Inputs

- `workflow.subagentDrivenDevelopment`
- enabled Codex and Claude clients
- required template references from Phase 13 specs `001` and `002`
- derived `effectivePermissions`
- existing workflow skill target contracts from `phase-03/004` and
  `phase-03/005`

## Outputs

| --- | --- | --- |

The exact generated Markdown for both targets is identical except for output
path and lockfile descriptor:

```markdown
---
name: subagent-driven-change
description: Use when a scoped implementation can be delegated to an implementation subagent and then independently reviewed for spec compliance before code quality.
---

<!-- Generated by Agent Profile Compiler. Do not edit by hand. -->

# Subagent-Driven Change

## Preconditions

Use this workflow only when the task has a clear spec, acceptance criteria, and file ownership. Keep tightly coupled or architectural decisions in the parent session unless the user explicitly asks for delegation.

Required subagents: `implementer`, `spec-reviewer`, and `code-quality-reviewer`.

## Fresh Context

Each subagent prompt must include the full task text, relevant spec excerpts, non-goals, acceptance criteria, file ownership, constraints, expected tests, and any command limits. Do not rely on hidden chat history or a previous subagent's memory.

## Flow

1. Dispatch `implementer` with one bounded task and the complete context it needs.
2. If `implementer` returns `BLOCKED` or `NEEDS_CONTEXT`, resolve that before continuing.
3. If `implementer` returns `DONE_WITH_CONCERNS`, read the concerns before review and decide whether to fix, narrow scope, or continue.
4. Dispatch `spec-reviewer` with the original task, relevant spec excerpts, changed files, and implementer report.
5. Fix or escalate every spec-review issue before requesting code-quality review.
6. Dispatch `code-quality-reviewer` only after spec review reports compliance.
7. Fix Critical and Important code-quality issues before handoff, or document why a finding is intentionally deferred.
8. Run the relevant tests, golden tests, and doctor/check commands required by the spec before final response.

## Status Values

Implementation worker status values: `DONE`, `DONE_WITH_CONCERNS`, `BLOCKED`, `NEEDS_CONTEXT`.

Spec reviewer status values: `COMPLIANT`, `ISSUES_FOUND`, `NEEDS_CONTEXT`.

Code-quality reviewer status values: `ACCEPTABLE`, `ISSUES_FOUND`, `NEEDS_CONTEXT`.

## Safety

- Do not ask subagents to commit, push, create branches, install dependencies, read secrets, contact production systems, or upload source unless the user explicitly requested that action.
- Do not accept an implementation report without reading reviewer findings.
- Do not run code-quality review before spec-compliance review passes.
- Keep generated files deterministic and lockfile-tracked.

## Output

Final handoff must list what changed, tests run, contract impact, security impact, remaining risks or TODOs, and whether the spec acceptance criteria are fully met.
```

## Contracts

- The skill is generated only when `workflow.subagentDrivenDevelopment: true`.
- The compiler reports a deterministic issue if the skill is requested without
  the three required template references.
- The skill is project-local only.
- The skill does not include scripts, references, assets, plugins, dynamic
  shell context, `allowed-tools`, or subagent runtime calls.
- The skill does not broaden permissions.
- The skill remains below the doctor 300-line warning threshold.
- Generated output uses UTF-8, LF line endings, no trailing whitespace, and
  exactly one trailing newline.

## Security Rules

- Do not include literal secrets or environment values.
- Do not instruct source upload, production access, dependency installation,
  unsafe auto-approval, or automatic commits.
- Do not tell compile or doctor to invoke subagents.
- Do not emit Tabnine workflow skills in this phase.

## Acceptance Criteria

- Codex golden output matches the exact Markdown in this spec.
- Claude golden output matches the exact Markdown in this spec.
- The skill is absent when `workflow.subagentDrivenDevelopment` is false or
  omitted.
- The compiler reports the documented deterministic issue when the workflow
  gate is true but required template references are missing.
- The skill includes fresh-context prompting rules.
- The skill enforces spec review before code-quality review.
- The skill includes exact status values for all three roles.
- Generated output contains no secrets, environment values, dynamic shell
  context, scripts, references, assets, tool pre-approval, production access,
  source upload, dependency auto-install, or automatic commit instructions.
