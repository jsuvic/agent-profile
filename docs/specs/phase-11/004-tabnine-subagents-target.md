# Spec: Tabnine Subagents Target

## Status

Implemented. Lifted from `phase-later/002-subagents-targets.md` on 2026-05-16.
Landed on 2026-05-17 in `dcf18bd` (PR #16). Depends on
`001-subagents-schema.md`.

This is the riskiest of the three targets in this phase because Tabnine
documents the feature as experimental and warns that custom subagents may
execute tools without per-action confirmation. The first implementation
restricts output to read-only subagents and does not write
`.tabnine/agent/settings.json`.

## Problem

Tabnine CLI documents project custom subagents under
`.tabnine/agent/agents/*.md`, but the feature is marked experimental and
custom subagents require explicit `experimental.enableAgents: true` in
`settings.json`. Without a target-specific spec, Agent Profile Compiler could
emit Tabnine subagents that execute shell, write files, or use the browser
agent — all behaviors currently documented as no-confirmation.

## Goal

Define the byte-level project-local generation contract for Tabnine subagents:
output path, frontmatter shape, mapping from the client-neutral schema in
`001`, the read-only-only first implementation, and the explicit decision to
defer `.tabnine/agent/settings.json` writes.

## Non-Goals

- implementing the target (separate code change)
- generating user-level (`~/.tabnine/agent/`), managed, admin, or plugin
  subagents
- generating Tabnine `run_shell_command`, `write_file`, browser-agent
  settings, extension references, or `.tabnine/agent/settings.json`
- generating `toolScope: workspace-write` Tabnine subagents until safer
  confirmation semantics exist or a later spec explicitly accepts the risk
- modifying Tabnine guideline output (owned by `phase-02/001`)
- modifying Tabnine MCP config output (owned by `phase-02/002`)
- defining doctor checks (owned by `005-doctor-subagent-checks.md`)

## User Flow

1. The user enables `capabilities.delegation.subagents` per
   `001-subagents-schema.md` with `clients.tabnine.enabled: true` and at
   least one `toolScope: read-only` agent intent.
2. The compiler produces `.tabnine/agent/agents/<name>.md` per intent.
3. The user must manually enable `experimental.enableAgents: true` in
   `.tabnine/agent/settings.json`. The compiler does not write that file in
   this phase.
4. The user starts Tabnine normally. The compiler never starts Tabnine and
   never invokes the subagent.
5. `agent-profile doctor` reports drift, unsafe broadening, and an
   informational finding when runtime enablement cannot be verified, per
   `005-doctor-subagent-checks.md`.

## Verified Target Surface

Verification date: 2026-05-15. Re-verify against current official Tabnine docs
before implementation begins.

- project path: `.tabnine/agent/agents/`
- user path exists but is out of scope
- Markdown files with YAML frontmatter
- required fields: `name`, `description`
- optional fields include `kind`, `tools`, `model`, `temperature`,
  `max_turns`, `timeout_mins`
- custom subagents require `experimental.enableAgents: true` in
  `.tabnine/agent/settings.json`
- project settings path is `.tabnine/agent/settings.json`
- the docs currently mark subagents experimental and warn that custom
  subagents may execute tools without per-action confirmation

## Generated Artifact Shape

For an intent named `code-reviewer` with `toolScope: read-only`,
`modelPreference: inherit`, `maxTurns: 10`, and `timeoutMinutes: 5`, exact
golden output is:

```markdown
---
name: code-reviewer
description: Use for focused code review before handoff.
kind: local
tools:
  - read_file
  - grep_search
max_turns: 10
timeout_mins: 5
---

<!-- Generated by Agent Profile Compiler. Do not edit by hand. -->

# Code Reviewer

Review changed code. Report only actionable findings with severity,
affected file or symbol, and the smallest safe remediation.
```

## Mapping

| Profile data                 | Tabnine output                                                                                                |
| ---------------------------- | ------------------------------------------------------------------------------------------------------------- |
| `name`                       | frontmatter `name` and filename `<name>.md`                                                                   |
| `description`                | frontmatter `description`                                                                                     |
| `prompt`                     | Markdown body after generated-file header, preceded by `# <Title-Cased name>`                                 |
| `toolScope: read-only`       | `kind: local` and `tools: [read_file, grep_search]`                                                           |
| `toolScope: workspace-write` | deferred: produce a deterministic not-generated or safety error result until Tabnine confirmation semantics change |
| `modelPreference: inherit`   | omit `model`                                                                                                  |
| `modelPreference: fast`      | omit `model` for MVP                                                                                          |
| `modelPreference: balanced`  | omit `model` for MVP                                                                                          |
| `modelPreference: capable`   | omit `model` for MVP                                                                                          |
| `maxTurns`                   | `max_turns`                                                                                                   |
| `timeoutMinutes`             | `timeout_mins`                                                                                                |

The first implementation must generate only read-only Tabnine subagents. It
must not emit `run_shell_command`, `write_file`, browser-agent settings,
extension references, or `.tabnine/agent/settings.json`.

The Markdown body title is derived deterministically from `name` (kebab-case
to Title Case with spaces).

## Output Contract

All generated artifacts must use:

- UTF-8
- LF line endings
- exactly one trailing newline
- no trailing whitespace
- stable frontmatter key order: `name`, `description`, `kind`, `tools`,
  `max_turns`, `timeout_mins`
- `tools` is emitted as a YAML block sequence; items are sorted lexicographically
- the generated-file HTML comment header on the line immediately after the
  closing frontmatter delimiter and one blank line

## Runtime Enablement Decision

Runtime enablement (`experimental.enableAgents: true` in
`.tabnine/agent/settings.json`) is not part of this phase. Reasons:

- The docs currently warn that custom subagents execute tools without
  per-action confirmation.
- A compiler-managed `settings.json` write would silently flip an
  experimental, no-confirmation runtime switch on the user's machine.
- The lockfile would have to take ownership of a settings file with broader
  scope than just subagents.

Doctor may report informational guidance when Tabnine subagent files exist
but `experimental.enableAgents` is not visible in project settings, per
`LINT-SUBAGENT-008` in `005-doctor-subagent-checks.md`.

A separate future spec (not part of Phase 11) may add a target-specific
settings write once safer Tabnine semantics exist or the project explicitly
accepts the risk.

## Contracts

- Generated subagents must narrow or preserve `effectivePermissions`; they
  must never broaden them.
- Tabnine subagents emit only read-only tools in this phase. Other tool sets
  require a future spec.
- `toolScope: workspace-write` produces a deterministic not-generated result
  or `unsafe_generated_content` compile issue while the feature remains
  experimental/no-confirmation. The compiler must not fall back to a
  read-only subagent silently.
- Existing user-authored `.tabnine/agent/agents/<name>.md` files must not be
  overwritten unless lockfile ownership proves they were generated by Agent
  Profile Compiler.
- This phase does not write `.tabnine/agent/settings.json`.
- Generated output is deterministic byte-for-byte for the same profile.
- The lockfile maps each output to `tabnine-subagents` and
  `targets/tabnine-subagents/<name>@1`.

## Security Rules

- Do not emit `run_shell_command`, `write_file`, or any write-capable tool
  while the feature remains experimental/no-confirmation.
- Do not emit browser-agent settings, extension references, or other
  privileged surfaces.
- Do not write `.tabnine/agent/settings.json` in this phase.
- Do not embed secrets, environment variable values, tokens, bearer headers,
  or production endpoints.
- Do not auto-install third-party MCP servers, plugins, extensions, or
  skills.
- Do not emit user, managed, admin, or plugin subagent files.
- Do not emit source-upload instructions or production access.

## Acceptance Criteria

- Output path is exactly `.tabnine/agent/agents/<name>.md` for each enabled
  intent.
- Frontmatter contains required fields and any optional fields produced by
  the mapping table in stable order.
- Body contains the generated-file header, a deterministic title, and the
  prompt text.
- `read-only` intents emit only `read_file` and `grep_search` in `tools`.
- `workspace-write` intents produce a deterministic not-generated or
  `unsafe_generated_content` compile issue while the feature remains
  experimental/no-confirmation.
- `.tabnine/agent/settings.json` is never written by this phase.
- Golden output for the `subagents-enabled` fixture matches exact bytes.
- Lockfile maps each output to `tabnine-subagents` and
  `targets/tabnine-subagents/<name>@1`.

- negative test asserts no secret-like values in generated output
- negative test asserts no environment variable values are emitted

  `browser_agent`) is at least warned per `005-doctor-subagent-checks.md`,
  with hyphen-vs-underscore normalization applied
