# Spec: Claude Subagents Target

## Status

Draft. Lifted from `phase-later/002-subagents-targets.md` on 2026-05-16. Not
approved. Depends on `001-subagents-schema.md`.

## Problem

Claude documents project subagents under `.claude/agents/*.md`. Without a
target-specific spec, Agent Profile Compiler could generate Claude subagents
that broaden tool access beyond `effectivePermissions`, use unsafe permission
modes, or emit fields that Claude treats as silently inherited.

## Goal

Define the byte-level project-local generation contract for Claude subagents:
output path, frontmatter shape, mapping from the client-neutral schema in
`001`, deterministic formatting, and the safety subset Phase 11 will support
in its first implementation.

## Non-Goals

- implementing the target (separate code change)
- generating user-level (`~/.claude/agents/`), plugin-level, managed, or
  admin-level subagents
- generating Claude hooks, memory, worktree isolation, background mode,
  inline MCP server definitions, or skills preload
- generating `permissionMode: bypassPermissions`
- modifying project Claude settings (owned by `phase-03/002`)
- modifying workflow skill output (owned by `phase-03/005`)
- defining doctor checks (owned by `005-doctor-subagent-checks.md`)

## User Flow

1. The user enables `capabilities.delegation.subagents` per
   `001-subagents-schema.md` with `clients.claude.enabled: true`.
2. The compiler produces `.claude/agents/<name>.md` for each agent intent.
3. The user starts Claude normally. The compiler never starts Claude and
   never invokes the subagent.
4. `agent-profile doctor` reports drift, unsafe broadening, and schema
   problems per `005-doctor-subagent-checks.md`.

## Inputs

- validated `AiProfile` with subagent intents per `001-subagents-schema.md`
- derived `effectivePermissions`
- compiler determinism contract from `phase-01/003-compiler-determinism.md`
- current official Claude subagent docs verified before implementation:
  `https://code.claude.com/docs/en/sub-agents`

## Outputs

| Field         | Value                                                   |
| ------------- | ------------------------------------------------------- |
| target id     | `claude-subagents`                                      |
| template id   | `targets/claude-subagents/<name>@1`                     |
| output path   | `.claude/agents/<name>.md` per agent intent             |
| fixture input | `fixtures/subagents-enabled/ai-profile.yaml`            |
| fixture gold  | `fixtures/subagents-enabled/expected/.claude/agents/<name>.md` |

The lockfile must map each generated output path to target id
`claude-subagents` and template id `targets/claude-subagents/<name>@1`.

## Verified Target Surface

Verification date: 2026-05-15. Re-verify against current official Claude
subagent docs before implementation begins.

- project path: `.claude/agents/`
- user path exists but is out of scope
- Markdown files with YAML frontmatter
- required frontmatter fields: `name`, `description`
- body is the subagent system prompt
- optional fields include `tools`, `disallowedTools`, `model`,
  `permissionMode`, `mcpServers`, `hooks`, `maxTurns`, `skills`,
  `initialPrompt`, `memory`, `effort`, `background`, `isolation`, `color`
- subagents inherit parent tool access by default unless restricted
- `permissionMode: bypassPermissions` bypasses prompts and must not be
  generated
- parent `bypassPermissions`, `acceptEdits`, and `auto` modes can override or
  ignore subagent frontmatter

## Generated Artifact Shape

For an intent named `code-reviewer` with `toolScope: read-only`,
`modelPreference: inherit`, `maxTurns: 10`, and the prompt body shown in the
`subagents-enabled` fixture, exact golden output is:

```markdown
---
name: code-reviewer
description: Use for focused code review before handoff.
tools: Read, Glob, Grep
model: inherit
permissionMode: plan
maxTurns: 10
---

<!-- Generated by Agent Profile Compiler. Do not edit by hand. -->

# Code Reviewer

Review changed code. Report only actionable findings with severity,
affected file or symbol, and the smallest safe remediation.
```

## Mapping

| Profile data                 | Claude output                                                                       |
| ---------------------------- | ----------------------------------------------------------------------------------- |
| `name`                       | frontmatter `name` and filename `<name>.md`                                         |
| `description`                | frontmatter `description`                                                           |
| `prompt`                     | Markdown body after generated-file header, preceded by `# <Title-Cased name>`       |
| `toolScope: read-only`       | `tools: Read, Glob, Grep` and `permissionMode: plan`                                |
| `toolScope: workspace-write` | omit `permissionMode`; emit write tools only when `effectivePermissions` allows it  |
| `modelPreference: inherit`   | `model: inherit`                                                                    |
| `modelPreference: fast`      | `model: inherit` for MVP unless Claude exposes a stable named tier in current docs  |
| `modelPreference: balanced`  | same as above                                                                       |
| `modelPreference: capable`   | same as above                                                                       |
| `maxTurns`                   | frontmatter `maxTurns`                                                              |
| `mcpServers`                 | string references only, after `phase-later/008` is approved                         |

The first implementation must not emit Claude `hooks`, `memory`, `isolation`,
`background`, `effort`, `color`, `initialPrompt`, inline MCP server
definitions, or `skills` preload. The Markdown body title is derived
deterministically from `name` (kebab-case to Title Case with spaces).

`tools` in `read-only` mode is an allowlist of the minimum surface; omitted
tool fields imply Claude's full inherited surface, which is forbidden for
generated read-only agents.

## Output Contract

All generated artifacts must use:

- UTF-8
- LF line endings
- exactly one trailing newline
- no trailing whitespace
- stable frontmatter key order: `name`, `description`, `tools`, `model`,
  `permissionMode`, `maxTurns`, `mcpServers`
- omit any optional frontmatter key whose value is unset rather than emitting
  an empty value
- the generated-file HTML comment header on the line immediately after the
  closing frontmatter delimiter and one blank line

## Contracts

- Generated subagents must narrow or preserve `effectivePermissions`; they must
  never broaden them.
- `toolScope: read-only` must not emit write, edit, shell, browser,
  dependency, network, production, or secret-capable tools.
- `toolScope: workspace-write` is allowed only when `effectivePermissions`
  permits workspace writes and must still keep shell, dependency
  installation, external network, secrets, and production access no looser
  than `effectivePermissions`.
- Existing user-authored `.claude/agents/<name>.md` files must not be
  overwritten unless lockfile ownership proves they were generated by Agent
  Profile Compiler.
- `permissionMode: bypassPermissions` must never be generated.
- The compiler must reject inline MCP server definitions in the agent intent
  before reaching this target.
- Generated output is deterministic byte-for-byte for the same profile.

## Security Rules

- Do not generate `permissionMode: bypassPermissions`.
- Do not generate Claude `hooks`, `memory`, `isolation`, or `background`
  fields.
- Do not embed secrets, environment variable values, tokens, bearer headers,
  or production endpoints.
- Do not emit source-upload instructions.
- Do not emit production access.
- Do not auto-install third-party MCP servers, plugins, or skills.
- Do not emit global, user, managed, admin, or plugin subagent files.

## Acceptance Criteria

- Output path is exactly `.claude/agents/<name>.md` for each enabled intent.
- Frontmatter contains required fields and any optional fields produced by the
  mapping table in stable order.
- Body contains the generated-file header, a deterministic title, and the
  prompt text.
- `read-only` intents emit only `Read, Glob, Grep` in `tools` and
  `permissionMode: plan`.
- `workspace-write` intents do not exceed `effectivePermissions`.
- `bypassPermissions` is never emitted.
- Golden output for the `subagents-enabled` fixture matches exact bytes.
- Lockfile maps each output to `claude-subagents` and
  `targets/claude-subagents/<name>@1`.

## Tests

- golden test asserts exact bytes for
  `fixtures/subagents-enabled/expected/.claude/agents/code-reviewer.md`
- frontmatter YAML round-trips through a strict parser without errors
- `read-only` mapping emits `tools: Read, Glob, Grep` and
  `permissionMode: plan`
- `workspace-write` mapping with `effectivePermissions.shell.run = "deny"`
  emits no shell tools
- absence-of-output test when `clients.claude.enabled: false`
- absence-of-output test when `capabilities.delegation.subagents.enabled` is
  false
- negative test asserts generated output never emits
  `permissionMode: bypassPermissions`
- negative test asserts generated output contains no `hooks`, `memory`,
  `isolation`, `background`, `effort`, `color`, or `initialPrompt`
- negative test asserts no secret-like values in generated output
- negative test asserts no environment variable values are emitted
- deterministic test asserts the same profile produces byte-identical output
  on repeated compile
- collision test asserts a name colliding with Claude built-ins
  (`explore`, `plan`, `general-purpose`) is at least warned per
  `005-doctor-subagent-checks.md`
