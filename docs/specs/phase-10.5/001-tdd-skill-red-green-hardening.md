# Spec: TDD Skill Red/Green Hardening

## Status

Verified. Belongs to Phase 10.5. Implemented on 2026-05-17 after explicit
implementation approval.

## Problem

The verified `tdd-change` workflow skill tells agents to add or update a
focused failing test first, but it does not make the RED and GREEN proof points
explicit enough for enforcement by the agent or review by the human.

The current generated skill also does not warn against common test-design
failures:

- assertions that verify mock objects rather than production behavior
- production methods added only to make tests easier
- broad mocks added before understanding dependency side effects
- partial mock objects that do not match the real data shape
- test suites that become mostly mock setup instead of behavior checks

Without those guardrails, an implementation can claim TDD while writing tests
after the code, watching a test pass immediately, or testing the test double
instead of the product behavior.

## Goal

Update the generated Codex and Claude `tdd-change` workflow skill so it
requires:

1. a focused failing test or golden fixture before behavior code changes
2. an explicit RED command and expected failure reason
3. a minimal GREEN implementation
4. an explicit GREEN command and passing result
5. a short anti-pattern gate for mocks and test-only production APIs
6. a report format that records what was proven

## Non-Goals

- changing the profile schema
- changing `sdd-change` or `final-review`
- generating skill `references/`, scripts, assets, plugins, MCP declarations,
  or subagents
- changing Tabnine guideline output
- requiring compile or doctor to run tests
- requiring commits, branch creation, worktree creation, or dependency
  installation
- deleting user-written implementation code when a test was not written first

## User Flow

1. A user enables `workflow.tdd: true` with Codex and/or Claude enabled.
2. `agent-profile compile --dry-run` previews the hardened generated
   `tdd-change` skill.
3. `agent-profile compile --write` writes the hardened project-local skill and
   lockfile descriptor.
4. During an implementation session, the agent loads the skill and records the
   RED command, RED failure reason, GREEN command, and GREEN result.
5. Final review can check whether the RED and GREEN proof points were reported.

The exact generated Markdown for both targets is identical except for output
path and lockfile descriptor:

```markdown
---
name: tdd-change
description: Use when changing behavior where a focused failing test or golden fixture must prove RED before implementation and GREEN after the minimal fix.
---

<!-- Generated by Agent Profile Compiler. Do not edit by hand. -->

# TDD Change

## Instructions

1. Identify the smallest observable behavior covered by the approved spec.
2. Add or update one focused failing test or golden fixture before changing behavior code.
3. Run the narrowest relevant test command and confirm RED: the test fails for the expected reason, not because of a typo, setup error, or unrelated failure.
4. Implement the smallest change that satisfies the failing test and the spec.
5. Run the same focused test command and confirm GREEN: the test passes without new warnings or unrelated failures.
6. Refactor only after GREEN, then rerun the focused test command.
7. Do not update golden files only to hide an unexplained behavior change.

## Testing Anti-Patterns

- Do not assert on mock elements or mock call counts when a real behavior assertion is possible.
- Do not add production methods, flags, or exports that exist only for tests.
- Do not mock a dependency until you understand the side effects the test needs.
- Keep test doubles structurally complete enough to match the real data shape consumed by the code.
- If mock setup is larger than the behavior under test, consider a narrower integration test or a simpler production boundary.

## Output

Report the RED command and expected failure, the GREEN command and passing result, any refactor rerun, and any TDD exception that required human approval.

## Safety

- Do not install dependencies automatically.
- Do not run broad or destructive commands without explicit approval.
- Do not include secrets, environment variable values, or production endpoints in tests or fixtures.
```

## Contracts

- Only `tdd-change` output bytes change in this phase.
- Template ids for `tdd-change` must bump from `@1` to `@2`; `sdd-change` and
  `final-review` keep their current template ids.
- The skill remains project-local only.
- The skill remains below the doctor 300-line warning threshold.
- The skill does not emit `allowed-tools`, dynamic shell context, scripts,
  references, assets, plugins, subagents, or MCP declarations.
- The skill does not grant permissions, execute commands at compile time, or
  install dependencies.
- Generated output stays byte-deterministic with LF endings and exactly one
  trailing newline.

## Security Rules

- Do not include literal secrets, environment variable values, production
  endpoints, source-upload instructions, dependency auto-install instructions,
  or unsafe auto-approval instructions.
- Do not instruct the agent to read `.env`, credentials, production data, or
  source outside the project.
- Do not instruct compile or doctor to execute tests.
- Do not use shell dynamic context injection in generated skill bodies.
