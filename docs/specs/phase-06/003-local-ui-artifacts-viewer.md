# Spec: Local UI Artifacts Viewer

## Status

Approved

## Problem

After running `agent-profile compile`, the user wants to inspect what was
generated and what each target sees, without opening every file in their
editor.

## Goal

Render the compiler's output as a tree on the left and a read-only file
preview on the right at `/artifacts`, matching the wireframe
`Artifacts — A — Tree + preview` variant. Each file shows a status badge:
`gen` (generated by the compiler), `manual` (present on disk but not
generated), or `drift` (generated previously but the on-disk content no
longer matches the lockfile).

## Non-Goals

- editing files in the browser
- writing files (deferred to CLI; Phase 5 owns the contract)
- a diff view (Phase 5 + Phase later own diff)
- the by-target columnar variant (`B`) — single variant in Phase 6

## User Flow

1. User clicks Artifacts in the sidebar.
2. The page calls the in-process compiler with the validated profile and
   the doctor for drift status.
3. The tree shows generated paths grouped by directory. Drift items show
   the `drift` badge.
4. User clicks a tree row; the right pane shows the file's path, status,
   owner target, hash, and the file's text content as preview.
5. The page surfaces the trust banner and a "preview is read-only" caption.

## Inputs

- `CompileResult` from `compileProfile`
- `DoctorResult` from `runDoctor`, used to detect drift items
- the project root (for relative-path display)

## Outputs

- a `/artifacts` route
- a tree component that groups files by their first path segment
- a preview pane showing path, status badge, hash, owner target, content
- a banner reminding the user that writes happen via the CLI

## Contracts

- the route does not invoke `applyWritePlan` or any write-capable function
- the tree is built from `CompileResult.files`; drift overlay is built
  from `DoctorResult.issues` filtered to `LINT-LOCK-*` codes
- if `compileProfile` returns `ok: false`, the page renders the issues
  list instead of a tree

## Security Rules

- never display content that matches `containsSecretLikeLiteral` — replace
  the matched range with `«redacted»` and add a banner above the preview
- never include the absolute filesystem path; relative paths only
- preview content is rendered as plain text (no HTML, no script execution)
- file size cap on preview: if a generated file is larger than 256 KiB,
  render a "preview truncated" message with the first 256 KiB

## Acceptance Criteria

- `/artifacts` shows the tree with generated, manual, and drifted states
- selecting a file shows its preview, hash, and owner target
- the trust banner appears
- "writes happen via the CLI" caption is visible above the preview
  actions area (no buttons in Phase 6)
- when no profile is present, the page renders an empty-state with a
  pointer to `agent-profile init`

## Tests

- snapshot of the tree built from a known fixture's `CompileResult`
- a test that asserts no `applyWritePlan` import is reachable from the
  client bundle
- a redaction test confirming a secret-like literal in generated content
  is masked

## Documentation Updates

- `apps/web/README.md` documents the `/artifacts` route and the read-only
  posture

## Final Review Checklist

- no write actions
- redaction applied
- drift surfaces inline, not buried
- tree groups by directory, sorted deterministically
