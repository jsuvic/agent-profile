# Spec: Import Ownership Lockfile Conformance (0.4.1 patch)

## Status

Approved 2026-07-07. Patch-scope conformance fix to the approved
phase-14/001 ownership proof order. Target release: 0.4.1.

Amended 2026-07-07 (approved same day) after the I1 spec review: the
parity contract exposed that `planRegionAwareWrites` has no manual-owned
branch and refuses such files as `unknown-ownership`, contradicting both
the behavior table and phase-14/001's own manual-owned rule ("Never
write. Report as preserved.") - and causing perpetual compile refusals
for users who chose preserve-as-manual at init. The amendment authorizes
exactly one minimal planner branch (see Compile Planner Amendment) and
extends parity coverage to the reviewer-identified gaps (lockfile v1
rows, no-entry rows).

Second amendment 2026-07-07: extending parity to no-entry rows as
blanket category equality overreached. For markerless no-lockfile-entry
files, init and compile diverge by design (phase-14/001's consent-gated
adoption: init is the only flow that may offer adoption; compile refuses
and points at init). Parity is therefore redefined: action-category
equality for lockfile-backed rows only; no-entry rows are pinned by the
binding expected-pair table below. No behavior change is authorized by
this amendment.

## Problem

Phase-14/001 fixes the ownership proof order: "1. lockfile v2 ownership
entry" wins before any content inspection. The compile write path honors
this (`planRegionAwareWrites` consults the lockfile first and refuses
hash-mismatched `generated-owned` files as drift). The import report does
not: `buildPhase14ImportReport` loads lockfile ownership - its own comment
says "Lockfile v2 ownership wins per the spec's ownership proof order" -
but applies it only to `client-config` entries. The `root-instructions`
branch classifies purely by content markers.

Observed consequence (reported 2026-07-07 from a real repository): for the
same `AGENTS.md`, `init --import` reports "legacy generated marker
detected; preserved as manual content" while `compile` refuses the same
file as "lockfile-owned generated region files that differ from
ai-profile.lock". The two commands assert contradictory ownership.

Latent hazard: following init's suggestion (`--strategy regions`) moves
compiler-owned generated bytes into the manual region; the next compile
inserts a fresh generated region beside them, duplicating instructions.

## Goal

`init --import` and `compile` derive ownership for root-instructions files
from the same proof order and give consistent verdicts for the same file
state. Lockfile-owned files are never offered regions adoption. Lockfile
drift is reported in the import report with a pointer to the compile
resolution path.

## Intent

Make the implementation match phase-14/001 as approved. No new ownership
semantics, no new flows; the drift intent-reconciliation feature is a
separate future spec.

## Non-Goals

- Intent reconciliation for hash mismatches (shared/client-specific/drift
  classification) - future phase-27/003.
- Existing-profile upgrade flow and lockfile provenance versioning -
  future phase-27/002.
- Any change to `planRegionAwareWrites` or compile refusal behavior
  beyond the single manual-owned branch authorized by the 2026-07-07
  amendment (the generated-owned and mixed paths were already
  conformant and stay untouched).
- Any change to lockfile schema, region markers, or generated artifacts.
- Changing the treatment of files with no lockfile entry (proof-order
  steps 3-5 stay byte-identical).

## Behavior (binding)

In `buildPhase14ImportReport`, for `root-instructions` entries with an
on-disk file, consult `ownershipByPath` before marker checks:

| Lockfile state | On-disk state | Report classification |
| --- | --- | --- |
| v2 `generated-owned` (or v1 whole-file entry, per proof-order step 2) | bytes match lockfile sha256 | ownership `generated-owned`, action `preserve`, note "lockfile-owned generated file; refresh via `agent-profile compile --write`". Never `insert-regions`, regardless of strategy. |
| v2 `generated-owned` (or v1) | bytes differ from sha256 | ownership `generated-owned`, action `preserve`, note "differs from ai-profile.lock (user edits or drift); `agent-profile compile` will refuse until resolved (`--force` overwrites)". Never `insert-regions`. Not counted as a conflict (compile enforces the hard stop). |
| v2 `mixed` | valid region markers | unchanged: `update-generated-region` (existing mixed path). |
| v2 `mixed` | markers missing or partial | `refuse-conflict`, note "lockfile records mixed ownership but region markers are missing or damaged; manual repair required". Counts as a conflict. |
| v2 `manual-owned` | any | ownership `manual-owned` (not `unknown`), action `preserve`. Never `insert-regions` - the lockfile records an explicit user declaration; converting it is out of scope. |
| no entry | any | unchanged: existing content-based flow (markers, legacy tag, strategy). |

### Compile Planner Amendment (2026-07-07, binding)

In `planRegionAwareWrites` (`apps/cli/src/index.ts`), when the lockfile
records `manual-owned` ownership for a region-aware path:

- Skip the write: the generated bytes for that path are not written.
- Emit no refusal; the run's exit status is unaffected by this path.
- Report the path as preserved (manual-owned) in the dry-run and write
  plan output.
- The rebuilt lockfile retains the `manual-owned` entry for the path -
  it is never dropped or reclassified by compile.

No other planner branch changes. This implements phase-14/001's
manual-owned default write behavior ("Never write. Report as
preserved."), which the planner previously approximated with an
`unknown-ownership` refusal. With this branch, the parity table's
manual-owned row is import `preserve` / compile `preserve`.

Wizard recommendation (`recommendStrategy`):

- The "legacy generated marker ... preserved as manual content" warning
  fires only when the file has no lockfile ownership entry. For
  lockfile-owned files the `generated-looking` tag may still be set, but
  the warning must not claim manual preservation.
- A drifted lockfile-owned root file adds a warning: "<path> differs from
  ai-profile.lock; compile will refuse until the drift is resolved."
- Lockfile-owned root files never contribute to the "can be adopted into
  mixed ownership" recommendation for the regions strategy.

## Inputs

`ai-profile.lock` (v1 and v2), on-disk root-instructions files, existing
import-report inputs. No new flags or config.

## Outputs

Corrected `Phase14ImportReport` classifications and notes; corrected
wizard intro/plan/warning lines for the affected states. Report content
changes only for previously-misclassified states (lockfile-owned or
lockfile-mixed root-instructions files); all other states byte-identical.

## Contracts

- Phase-14/001 ownership proof order is now enforced in the import
  report; that spec's text is unchanged.
- Verdict parity (binding, as redefined by the second 2026-07-07
  amendment):
  - Lockfile-backed rows (v2 `generated-owned`/`mixed`/`manual-owned`
    and v1 whole-file entries, match and drift): the import report's
    action and `planRegionAwareWrites`' verdict fall in the same
    category (update / preserve / refuse) - proven by a table-driven
    test, not review.
  - No-lockfile-entry rows: pinned by this expected-pair table instead
    of category equality (both columns assert current behavior; no code
    change):

    | On-disk state (no entry) | Import action | Compile verdict |
    | --- | --- | --- |
    | markerless, strategy preserve | `preserve` | refuse `unknown-ownership` with the adoption pointer |
    | markerless, strategy regions | `insert-regions` | refuse `unknown-ownership` with the adoption pointer |
    | all region markers valid | `update-generated-region` | region update (no-lockfile adoption path) |
    | partial markers | `refuse-conflict` | refuse `partial-markers` |

    The divergent markerless rows are phase-14/001's consent-gated
    adoption design: compile never adopts; init is the adoption flow,
    and compile's refusal message points at it.
  - In all rows the compile verdict is derived honestly (refusals
    detected via exit code/refusal output, never inferred from
    unchanged bytes).
- Report shape (field names, types) unchanged; only classification
  values and note strings for the affected states change. `init --json`
  output changes for those states only; this is the bug fix, documented
  in the CHANGELOG as behavior-correcting.
- No generated artifact, golden fixture, or compile output changes.

## Security Rules

- No new write paths; the fix only reclassifies reporting.
- The duplication hazard (generated bytes migrating into manual regions
  of lockfile-owned files) is closed by construction: `insert-regions` is
  unreachable for lockfile-owned paths.
- No secrets read; no network; no new dependencies.

## Acceptance Criteria

1. Parity table test passes per the redefined contract: category
   agreement for every lockfile-backed row (v2 rows, v1 match and
   drift, manual-owned via the amended planner branch - including the
   reported contradiction state: legacy marker + `generated-owned` +
   drifted bytes), and exact expected-pair assertions for the four
   no-entry rows (including compile's refusal reason and adoption
   pointer for the markerless rows). Compile verdicts are derived from
   actual refusal signals, never inferred from unchanged bytes.
2. A markerless lockfile-owned `AGENTS.md` under `--strategy regions` is
   reported `preserve` with the refresh note; `insert-regions` does not
   appear for it in the report or wizard plan.
3. Lockfile `mixed` + missing markers -> `refuse-conflict`; the conflict
   count increments and the wizard lists it.
4. Drift on a lockfile-owned file produces the drift note and the
   recommendStrategy warning; the "preserved as manual content" wording
   no longer appears for lockfile-owned files.
5. Every no-lockfile-entry scenario is byte-identical to current output
   (existing phase-14 tests and goldens pass unmodified).
6. Released as 0.4.1 with a CHANGELOG entry describing the corrected
   behavior.
7. (Amendment) A `manual-owned` lockfile entry for a region-aware path:
   `compile --write` succeeds, leaves the file byte-identical on disk,
   reports it preserved, keeps the lockfile entry `manual-owned`, and
   emits no refusal - in both dry-run and write modes.
