# Spec: Dispatcher Follow-Up Offers + Doctor and Upgrade Clarity

## Status

Approved 2026-07-12. Synthesized from the second field test (macOS,
0.4.4): a fresh user routed to doctor got 13 raw errors and an exit,
and the upgrade choices were unexplained. Amends ADR 0014
(consent-gated follow-up chain; the no-loop-back-menu decision stands).

## Problem

Four findings from a real first-contact session:

1. The dispatcher runs one action and exits. Doctor printed 13 errors
   with per-issue command advice, then ended - the user must re-run
   `agent-profile` and choose again for every step of a multi-step
   recovery.
2. Doctor's 13 errors were really 3 root causes (10x foreign skills at
   generated paths; missing artifacts + missing lockfile; one
   legacy-marker file). Nothing summarizes issues into deduplicated
   recommendations.
3. Upgrade's choices carry no explanation. Worse, "Adopt recommended"
   is mislabeled: the implementation adopts the ENTIRE offered list
   (`selected = offered`) - no recommended subset exists anywhere. And
   nothing says that adopting edits `ai-profile.yaml` only, with files
   generated by the next `compile --write`.
4. The dispatcher menu prints the doctor wordmark
   (`formatLogo("doctor", ...)` hardcoded) before any choice is made,
   then the routed command prints its own - wrong label and a
   duplicate.

## Goal

A first-contact user can go from a broken repo to a healthy one inside
a single `agent-profile` invocation, consenting step by step, with
every choice explained in the menu itself.

## Intent

Presentation and flow-guidance only: no new commands, no schema or
artifact changes, no new frozen surfaces; every mutation keeps its
existing consent gate.

## Decision Rules

1. Chain doubt -> offer, never auto-run: each follow-up action requires
   its own explicit confirmation; decline always exits cleanly.
2. Summary doubt -> group by the command that fixes it; per-issue
   detail stays available below the summary, never replaced.
3. Label doubt -> the label states what the code does ("Adopt all
   available"), not what a design once intended; scripted flag names
   are compatibility surface and keep their spelling.
4. Frozen doubt -> non-interactive doctor/upgrade text, `--json`, exit
   codes, and the `--adopt-recommended` flag are untouched.

## Non-Goals

- A loop-back action menu (ADR 0014's rejection stands; the follow-up
  is a single-next-action confirm, not a menu).
- Auto-running anything without a per-step confirmation.
- A "recommended subset" feature for upgrade (a catalog `recommended`
  flag is a possible future addition; this patch makes the label
  honest instead).
- Renaming the `--adopt-recommended` CLI flag (kept for
  compatibility; help text documents it as adopt-all).
- Any change to doctor's issue detection or upgrade's insertion
  machinery.

## Behavior (binding)

### 1. Dispatcher follow-up offers (amends ADR 0014)

After a routed action completes inside the dispatcher (interactive TTY
only), re-evaluate the repository state with the same imported
machinery and offer the next action as a confirm:
"Next: Generate agent files (`compile --write`). Run it now?" -
default No. Accept -> run it (with its own flow and consent defaults),
then offer again. Decline or reaching "current" -> exit. The process
exit code is the last completed action's code (a declined offer does
not alter it). Direct subcommand invocations (`agent-profile doctor`
etc.) gain no follow-up - this is dispatcher-only.

Consumed-state filter (amended 2026-07-12 from the approval review):
"offered at most once" is a filter on the evaluation, not a stop
trigger. The dispatcher tracks the states consumed this invocation
(offered and run); on each re-evaluation it drops consumed states from
the applicable set and offers the highest-priority REMAINING action.
The chain stops with a note only when the filtered set is empty and
the state is not "current". This matters because doctor is read-only:
the Broken state necessarily survives a doctor run, and a literal
re-evaluation would re-select doctor (or dead-end on a stop trigger)
instead of progressing - the field-log chain (doctor -> compile
--write) is only reachable under the filter semantics. Offering a
mutating action while a broken state persists is safe by the product's
existing defense in depth: every offered command keeps its own guards,
previews, and refusal paths (e.g. compile refuses damaged-marker files
with guidance), so the worst case of a premature offer is a clean,
explained refusal - never a harmful write. The doctor recommendation
summary (behavior 2) may supply the offer's wording where a grouped
recommendation maps to the offered action, but the routing decision
itself always comes from the filtered state evaluation - one routing
mechanism, not two.

### 2. Doctor recommendation summary (interactive TTY only)

After the existing per-issue output, doctor prints a summary that
groups actionable issues by their fixing command / root cause,
deduplicated, with counts - e.g. "3 recommendations: (1) 10 foreign
skills at generated output paths -> review and adopt or overwrite;
(2) missing generated artifacts and lockfile -> `agent-profile compile
--write`; (3) CLAUDE.md carries the legacy marker -> `agent-profile
init --import --strategy regions --write`". Grouping keys off each
issue's existing guidance text/code - no new detection. Non-interactive
doctor output and `--json` stay byte-identical. The dispatcher's
follow-up offer reuses this grouping where a recommendation maps to a
dispatcher action.

### 3. Upgrade clarity

- Relabel the interactive option "Adopt recommended" ->
  "Adopt all available"; add hints to all three options:
  - Keep current: "change nothing and exit"
  - Adopt all available: "add every listed capability to
    ai-profile.yaml"
  - Customize: "choose which capabilities to add"
- The Available-capabilities note gains one closing line: "Adopting
  adds entries to ai-profile.yaml only; run `agent-profile compile
  --write` afterward to generate the files."
- `--adopt-recommended` keeps its spelling; `--help` describes it as
  adopting all offered capabilities.

### 4. Dispatcher menu logo

The menu prints a neutral `agent-profile` wordmark (no command label);
the routed command's own logo prints exactly once, when it runs.

## Contracts

- Frozen: non-interactive doctor and upgrade text, `--json` shapes,
  all exit codes, the `--adopt-recommended` flag, non-TTY bare help.
- The follow-up chain never runs an action without a fresh explicit
  confirmation, and every offered action retains its own internal
  consent gates (previews, write confirmations).
- The doctor summary adds lines after existing interactive output;
  nothing is removed or reordered.
- Interactive text changes are behavior-corrective; affected goldens
  (if any interactive fixtures exist) are enumerated in the CHANGELOG.

## Security Rules

- No auto-execution: consent per step, decline always available and
  clean.
- The state re-evaluation between offers is read-only.
- No new dependencies, commands, or network.

## Acceptance Criteria

1. Dispatcher scenario test (the field-log shape): broken repo ->
   doctor runs (read-only; Broken state persists) -> the consumed-state
   filter drops Broken and the follow-up offers the next applicable
   action (compile --write) -> accept -> runs -> next offer or current
   -> chain ends; each step confirmed; declining at any point exits
   with the last completed action's code. This criterion is only
   satisfiable under the filter semantics - it doubles as the
   regression test for the doctor-cannot-progress contradiction.
2. Consumed-state filter: a consumed state is never re-offered even
   when it persists after its action; the run ends with a note only
   when no un-consumed applicable action remains and the state is not
   "current" (negative fixture: an action that leaves every remaining
   applicable state consumed).
3. Doctor (interactive) prints the grouped recommendation summary with
   deduplicated commands and counts; non-interactive doctor and --json
   are byte-identical to current output.
4. Upgrade menu shows the new label and all three hints; the
   capabilities note carries the compile pointer line; --help documents
   --adopt-recommended as adopt-all; the flag's behavior is unchanged.
5. The dispatcher menu shows the neutral wordmark; routing to doctor
   prints the doctor logo exactly once (regression on the double-logo).
6. Direct `agent-profile doctor` / `compile` / `upgrade` invocations
   gain no follow-up offer (frozen behavior).

## Tests

- Dispatcher chain tests via injected prompts: accept-chain,
  decline-exit, offered-once bound, exit-code passthrough of the last
  action.
- Doctor summary unit tests over a fixture issue list reproducing the
  field log (10x LINT-OWN-001 + LINT-STRUCT-003 + LINT-LOCK-001 +
  LINT-OWN-002 -> 3 grouped recommendations); non-interactive
  byte-identity golden.
- Upgrade prompt tests: label, hints, note line; flag-behavior
  regression (scripted --write --adopt-recommended unchanged).
- Logo: menu wordmark assertion + single doctor logo on routing.
